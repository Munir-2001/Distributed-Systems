PART I

Foundations

1Replicated state machine

The “database” is an array of 100 ints, positions[] — one copy per replica. The entire project rests on a single invariant: replicas that start identical and apply the same updates in the same order remain identical. Every other mechanism (quorums, epochs, elections) exists only to preserve “same updates, same order” while replicas crash.

The spec’s four TOB properties are not slogans — each one is carried by a specific mechanism:

Say at the viva“This is fixed-sequencer TOB. Its weakness — the sequencer is a single point of failure — is exactly what the election protocol repairs.”

3The two-phase update protocol

Why two phases? Applying on UPDATE receipt would be fatal — the coordinator could die having reached one replica, which would then hold state nobody else ever saw. Phase 1 makes the update stable (known to a majority) before anyone applies. Not 2PC: no unanimous vote, no abort path — it’s the commit round of Paxos / Viewstamped Replication with a majority.

Pipelining is safe: FIFO channels ⇒ each replica receives and ACKs UPDATEs in order ⇒ the quorum for seq s necessarily completes before s+1 ⇒ commits happen in order.

4Why a majority? Quorum intersection

The safety chain, four steps you should recite fluently: (a) u applied anywhere ⇒ u committed ⇒ a majority Q₁ observed u; (b) the spec guarantees a live majority Q₂ at election time; (c) Q₁∩Q₂ ≠ ∅ ⇒ some live replica observed u; (d) the election picks the most-recent-observed live replica ⇒ the winner observed u ⇒ it completes u and ships it to everyone. Nothing committed can be lost.

Hidden lemma: “has the most recent” implies “has everything before it” — FIFO channels + in-order commits mean you cannot observe seq s without having received all of ≤ s first. Without FIFO, “most recent” would be meaningless.

5⟨epoch : seq⟩ — the total order itself

epoch = a coordinator’s reign number — a Raft term, a Paxos ballot, a VR view. It only moves at elections: the winner sets epoch = maxSeen + 1. seq = position within the reign, reset to 0 each epoch. Update ids compare lexicographically — that ordering is the total order every replica applies.

6Sequential consistency — what clients may see

Reads are answered locally and immediately from applied state — no coordination. So a read may be stale (that replica hasn’t applied the newest update yet). What is forbidden is going backwards: each replica applies the one total order prefix-by-prefix, so values at any single replica move monotonically forward. That is sequential consistency; linearizability would require coordinated reads and is deliberately traded away for fast local reads.

PART II

Failures

7The failure model: crash-stop

Replicas fail by crashing silently and never recover — no Byzantine behavior. A crash is a behavior switch, not actor death: crashNow() cancels all timers and swaps in a receive loop that ignores everything. Typed crash points let the instructors kill a replica at protocol-critical instants — including mid-broadcast (the coordinator dies having reached only some replicas), which manufactures exactly the situation recovery must repair.

8Failure detection — three watchdogs + client timeouts

Asynchronous systems cannot distinguish slow from dead (the FLP obstacle). The spec assumes detection is accurate — effectively a perfect failure detector — and our job is sizing timeouts so the assumption holds: everything derives from coordinatorTimeoutDelay = 2×beat + maxLatencyPlusTolerance. 9Ring-based coordinator election

A modified Chang–Roberts ring over sorted ids. The Election token accumulates ⟨id → mostRecentObserved⟩ from every node it visits; when it reaches a node already in its candidate map it has completed a full lap — it now contains every reachable replica — and the decision is taken: freshest observation wins, higher id breaks ties. Replicas may crash mid-election: every hop must be ACKed; a missing ElectionAck within the timeout means the successor is dead — skip it. If the decided winner itself died, an overall timer restarts the whole election (liveness backstop only).

10Takeover & synchronization — a view change

The winner’s first acts, in order (Viewstamped Replication calls this a view change): (1) new regime — epoch = maxSeen+1, seq = 0; (2) complete interrupted updates — apply everything observed-but-unapplied, in id order (the spec’s safety clause); (3) broadcast SYNCHRONIZATION with its full history — followers apply what they miss (idempotent) and purge pending updates the log doesn’t vouch for; (4) only then heartbeat and serve new writes, while origins resubmit unacknowledged client writes.

Say at the viva“The purge is the dual of completion: the sync log is the authoritative start of the new epoch — pendings it vouches for get applied, pendings it doesn’t are dead-epoch leftovers whose origins resubmit them. Keeping them would let a later election win re-apply superseded updates.”

11Regime fencing — keeping dead coordinators dead

Channels have independent random delays, so a superseded coordinator’s messages can arrive after the new one’s. Every regime-bearing message is checked against the pair ⟨epoch, coordinatorId⟩, compared lexicographically: stale heartbeats are ignored, a delayed SYNCHRONIZATION can’t regress the epoch, old-epoch UPDATEs are neither observed nor ACKed. Equal-epoch ties resolve to the higher id — deterministic convergence instead of split-brain flip-flopping. Same idea as Raft’s “reject older terms”.

PART III

Clients, network, actors

12Exactly-once client semantics

Request ids are per-client counters, so the origin keys outstanding writes by the pair ⟨client, rid⟩ — two clients writing through the same replica both start at rid 0 and must not collide. The reply is claimed atomically (remove(key) != null → reply once) inside applyUpdate, so an update re-learned via a sync log can’t answer twice; a write lost with its coordinator is resubmitted and commits under a fresh id — the client sees exactly one result either way. Resubmission walks a LinkedHashMap, preserving each client’s send order across failover.

13The emulated network — FIFO is load-bearing

One NetworkChannel actor per (sender → destination) pair queues messages and delivers them one at a time with a random delay in [min, max] — so latency is random but order within a channel is preserved, and the original sender survives delivery. Three proofs silently lean on FIFO: (1) apply order — WRITEOK(s) can’t overtake UPDATE(s); (2) pipelining safety — per-replica ACK order ⇒ commit order; (3) sync-before-traffic — the winner’s SYNCHRONIZATION precedes its heartbeats and updates on the same channels. But FIFO holds per channel only — cross-sender ordering doesn’t exist, which is exactly why regime fencing (#11) is needed.

14Actor-model mechanics

One message at a time per actor — all replica state is lock-free because the protocol code is sequential per replica; concurrency exists only between actors. Messages are immutable (final fields; collections defensively copied and unmodifiable — the same instance is broadcast and retained for retransmission). Time is a message: every timeout is a scheduled self-message with a Cancellable handle, so even timers flow through the mailbox. Crash = behavior switch via become(). The abstract classes are the observability contract: the callbackOn* methods forward protocol events to test probes, which is why calling them at exactly the right moments is mandatory.

PART IV

Synthesis

15Safety vs. liveness — the closing argument

Safety (no lost committed update, no double apply, no two regimes) rests only on counting and comparing — quorum intersection, ⟨epoch:seq⟩ order, ⟨epoch, coordinatorId⟩ fencing, appliedIds — never on clocks. Liveness (writes commit, elections finish) is where timeouts, heartbeats, ring skipping and the restart backstop live, protected by the spec’s assumptions: a majority never crashes, detection is accurate, channels are reliable and FIFO.

Assumption (spec)What breaks without it
Strict majority never crashesno quorum to commit or elect — the system halts (by design, rather than diverging)
Accurate crash detectionfalse suspicion → election against a live coordinator; fencing bounds the damage deterministically
Reliable FIFO channelsgap-freedom lemma dies — “most recent” no longer implies “has everything”
Crash-stop (no recovery)a recovered replica with stale state would need a re-join/catch-up protocol (not required here)

?Viva drills — likely questions

Why two phases? Why not apply as soon as the UPDATE arrives?

Applying on UPDATE would let a replica apply something known only to itself (coordinator dies after reaching one replica). Phase 1 makes the update stable — observed by a majority — before Phase 2 permits anyone to apply. Once a majority has it pending, quorum intersection guarantees any future election winner knows it, so applying is safe.

Why exactly ⌊N/2⌋+1? Why not fewer, or all?

Majority is the smallest quorum where any two quorums intersect — that intersection carries committed updates across coordinator changes. Fewer (no intersection guarantee) loses updates; all (2PC-style) blocks on a single crash. Majority is the sweet spot: tolerates ⌈N/2⌉−1 crashes and never blocks while a majority lives.

Walk me through what happens if the coordinator crashes right after sending WRITEOK to one replica.

That replica applies; the others hold the update pending and their WRITEOK watchdogs fire → election. Every live replica observed the update (they ACKed it), so the freshest-observation criterion (tie → highest id) elects someone who has it. The winner completes it via completePendingUpdates, ships it in the SYNCHRONIZATION log, everyone converges — the applied write is never lost (uniform agreement). SIM F is exactly this.

Why does the election pick “most recent OBSERVED” rather than “most recent APPLIED”?

Commitment only guarantees a majority observed the update as pending — possibly nobody applied it (coordinator died before any WRITEOK left). If we elected by applied state we could lose a committed update. Observed = applied ∪ pending is the correct notion of “knowledge” for the intersection argument.

Can two coordinators exist at the same time?

Under the spec’s accurate-detection assumption, no — a coordinator is only replaced after actually crashing. Defensively, regimes are fenced by ⟨epoch, coordinatorId⟩ compared lexicographically: messages of a strictly smaller pair are ignored, and equal-epoch ties deterministically resolve to the higher id, so even a pathological double-win converges instead of flip-flopping.

What if a replica crashes during the election itself?

Every ELECTION hop must be ACKed. A dead successor never ACKs (crash fires before acking), the forwarder times out and skips to the next ring member. If the decided winner died, the overall election timer (4× coordinator timeout) restarts the election; the dead node no longer appears among candidates. SIM G shows a double skip live.

Is this linearizable? What consistency do clients actually get?

No — reads are local and can be stale. The guarantee is sequential consistency: all replicas apply the one total order, so per-replica reads are monotone and all clients agree on the order of writes. Linearizability would need coordinated (e.g., quorum) reads; the spec trades that for one-hop reads.

How do you guarantee a client write is never executed twice after a failover?

Two dedup layers. Update level: appliedIds makes application idempotent. Request level: the origin tracks writes by ⟨client, rid⟩ and only resubmits those still unclaimed after applying the sync log — if the write survived (in the log), applying it claims and answers it, so nothing is left to resubmit; if it didn’t survive, the stale pending entry was purged and the resubmission is the only copy.

Why must the resubmission order be preserved? (Why LinkedHashMap?)

Sequential consistency promises per-client program order. Pending writes W1, W2 from one client resubmitted inverted would commit as W2, W1 — a client that wrote A then B could end with A as the final value. Insertion order of the map = mailbox processing order = client send order, so walking it preserves program order across failover.

What does the WRITE-forward timeout watch exactly?

Per spec: that the coordinator initiates the broadcast in time. It’s armed when the origin forwards, and satisfied the moment the origin receives the UPDATE for its own request — from there the per-update WRITEOK watchdog takes over. Two watchdogs, two distinct failure windows.

What happens on epoch change to the sequence numbers, and why is that safe?

The winner sets epoch = maxSeen+1 and resets seq to 0; ids compare epoch-first, so ⟨1:0⟩ > ⟨0:57⟩ — the total order continues without collision. Uncommitted ⟨0:*⟩ leftovers are purged at synchronization, so they can never be applied after the regime change.

Which parts are safety and which are liveness? What breaks if a timeout is too short?

Safety: quorums, id ordering, fencing, dedup — timing-independent. Liveness: detection and elections. A too-short timeout violates the accurate-detection assumption (false suspicion): the system would run a spurious election against a live coordinator — availability noise, while fencing keeps the outcome deterministic. That’s why every timeout is derived from max latency + tolerance and the heartbeat interval.

Known limitations you’d state upfront?

(1) One in-flight election token per replica — overwritten ACK-retry state can drop a token; backstopped by the overall restart timer (liveness-only). (2) A decided token addressed to a dead winner circulates until that same backstop. (3) Unbounded history and full-history SYNCHRONIZATION — fine at project scale; production would checkpoint + send deltas. All documented in REPORT_NOTES.md.
